Voici un cahier des charges structuré pour le site web interactif du tableau périodique, compilant les objectifs et exigences du projet :
Public cible : Grand public, du collégien curieux de sciences à l’adulte amateur de culture scientifique. Le site doit être compréhensible par des non-spécialistes, tout en étant suffisamment intéressant pour des connaisseurs. On vise aussi bien une utilisation éducative (étudiants, enseignants) qu’une consultation loisir par toute personne intriguée par un élément.
Objectif général : Proposer une visualisation interactive du tableau périodique qui soit à la fois éducative et ludique. L’utilisateur doit pouvoir explorer les éléments de manière intuitive, apprendre des faits scientifiques fiables, tout en passant un bon moment grâce à un ton engageant et des surprises (anecdotes amusantes, illustrations).
Contenu par élément : Pour chaque élément, fournir un mini-dossier composé de :
Une courte description (quelques phrases) présentant l’élément (par ex. sa découverte, ses propriétés marquantes ou sa place dans la vie quotidienne).
2-3 usages typiques de l’élément (usages industriels, domestiques, biologiques, etc.), présentés sous forme de puces ou d’icônes.
Une anecdote scientifique amusante liée à l’élément – cela peut être une curiosité historique, une propriété étonnante, ou un fait moins connu mais divertissant. Le ton ici peut être léger, voire humoristique, tout en restant exact.
Des détails supplémentaires dépliables : une section “en savoir plus” que l’utilisateur peut afficher pour voir des informations plus techniques (par exemple : propriétés physiques précises, isotopes notables, précautions de sécurité, etc.). Cette section contiendra des données plus pointues pour les utilisateurs qui le souhaitent, sans encombrer la vue initiale.
Fonctionnalités clés :
Exploration par clic : L’utilisateur doit pouvoir cliquer (ou toucher) n’importe quelle case du tableau périodique pour voir apparaître les informations de l’élément correspondant. Cela peut se faire via une navigation type “page détail” ou une fenêtre modale superposée.
Recherche rapide : Une barre de recherche accessible permet de trouver un élément par son nom (ex: “Carbone”), son symbole (“C”) ou son numéro atomique (“6”). La recherche se met à jour en temps réel et propose des suggestions auto-complétées pour corriger d’éventuelles fautes de frappe.
Filtres dynamiques : Un panneau de filtres permettra de restreindre l’affichage à certains sous-ensembles. Les filtres envisagés : par catégorie chimique (ex: métaux, non-métaux, métalloïdes…), par famille (groupe du tableau, ex: halogènes), par état à température ambiante (solide/liquide/gaz), par propriété particulière (radioactifs, synthétiques, abondants/résiduels…). En activant un filtre, les éléments correspondants seront mis en avant (ou les autres masqués) de manière visible.
Design responsive fluide : Le tableau doit rester navigable sur mobile. Sur écran tactile, l’utilisateur pourra pincer pour zoomer sur le tableau, ou faire défiler pour voir les colonnes débordantes. Alternativement, un mode d’affichage listant les éléments par numéro pourrait être proposé sur mobile en complément. L’objectif est qu’aucune information ne soit cachée ou illisible sur petit écran.
Internationalisation (i18n) : Le site intègre dès le départ le support de deux langues, français et anglais. Un commutateur (ex: drapeau ou code “FR/EN”) permettra de passer de l’une à l’autre. Tous les textes de l’UI, ainsi que les contenus des éléments, doivent donc être prévus dans les deux langues. (Dans un premier temps, on peut remplir la version française et n’utiliser l’anglais qu’en interface, mais la structure doit permettre d’ajouter la traduction anglaise du contenu facilement.)
Animations fluides : L’interface comportera des animations pour rendre l’expérience plus vivante. Par exemple, au survol d’un élément, on peut le surligner ou afficher une info-bulle préliminaire. Lorsqu’on clique pour ouvrir la fiche d’un élément, l’apparition de la fiche sera animée (transition en fondu, glissement, ou carte qui se retourne façon flip). Ces animations doivent renforcer la compréhension (indiquer clairement ce qui se passe) et le plaisir d’utilisation, sans ralentir ni gêner la navigation.
Technologies imposées :
Framework : Svelte (version vanilla, sans SvelteKit). On utilisera Svelte pour construire une application web réactive monopage (SPA).
Langage : TypeScript pour la robustesse (détection d’erreurs de types, autocomplétion, etc.).
UI Library : Tailwind CSS pour accélérer le développement du design (utilisation d’utilitaires de style directement dans les composants). En complément, possible utilisation de shadcn/ui pour des composants pré-stylés (même si conçu à l’origine pour React, on peut s’inspirer de leurs principes d’UI et reproduire des composants équivalents en Svelte, par exemple pour les boutons, onglets, tooltips, etc., en utilisant Tailwind).
Animations : GSAP (GreenSock) ou Framer Motion. GSAP, très puissant en JS pur, pourra être appelé dans les hooks Svelte (onMount par ex.) pour orchestrer des animations complexes. Framer Motion a une version pour React, donc probablement GSAP conviendra mieux ici, sauf si on trouve une librairie équivalente pour Svelte. Dans tous les cas, l’animation sera codée en JS/TS au sein de composants Svelte.
Structure : Étant sans SvelteKit, le site reposera sur un seul fichier HTML faisant appel au bundle JS/CSS généré. On pourra utiliser un router JS pour gérer les vues (par ex. Routify ou un simple mécanisme maison basé sur l’URL hash ou History API).
Outils de build : Vite ou Rollup (généralement le template Svelte utilise Rollup ou Vite par défaut) pour transpiler le code TS et assembler le bundle, avec configuration pour Tailwind (postCSS) et minification en production.
Contraintes de livraison : La version initiale sera livrée en français entièrement fonctionnelle, avec l’infrastructure prête pour l’anglais (fichiers de langue, etc.). Le code source sera propre et bien documenté pour faciliter les évolutions. On privilégie une architecture front-end statique (pas de backend requis, hormis éventuellement servir le fichier JSON des éléments). Le site devra être hébergeable facilement (par ex. sur GitHub Pages, Vercel, Netlify…) et charger rapidement même avec la richesse de contenu (on fera attention au poids des assets comme les images ou les bibliothèques JS ajoutées).
Ce cahier des charges servira de référence tout au long du développement pour garder le cap sur les objectifs et s’assurer que toutes les fonctionnalités attendues sont bien mises en œuvre
